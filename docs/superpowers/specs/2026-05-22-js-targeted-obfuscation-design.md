# JS 定向压缩处理设计

## 背景

当前仓库中已经新增了一个 `JS 代码混淆` 模块，但现有实现采用的是“整段源码统一混淆”的能力，这与最新需求不一致。最新需求要求：

1. 整体源码只做压缩输出。
2. 不做整文件级别的变量名、控制流、属性名混淆。
3. 仅对用户指定的“特征元素”做定向处理。
4. 特征元素由页面表单输入，而不是写死在构建代码中。
5. 第一版特征元素范围仅包含：
   - 变量名
   - 字符串字面量

因此，该模块需要从“通用整段混淆工具”调整为“定向改写 + 压缩输出工具”。

## 目标

为 `/tools/js-obfuscator` 提供一个三段式工作台：

1. 左侧输入原始 JS 源码。
2. 中间通过表单输入“变量名”和“字符串字面量”两组特征元素。
3. 右侧输出“只压缩整体、仅定向处理指定元素”的结果代码。

用户可以在不手写 JSON 的前提下，通过表单精确控制哪些变量名和字符串会被改写；未列入配置的变量名和字符串保持原样。

## 非目标

以下能力明确不在本次范围内：

1. 不支持函数名定向处理。
2. 不支持对象属性名定向处理。
3. 不支持控制流平坦化、自保护、字符串数组拆分等整段混淆能力。
4. 不支持从文件读取配置，也不支持上传配置文件。
5. 不展示“命中数量”“改写差异视图”“历史记录”等增强能力。

## 页面结构

页面保持深色、工具化语气，但从现有双栏工作台改为三段式布局：

1. 左侧：源码输入区
   - 大面积代码输入框。
   - 用于粘贴或编辑原始 JS 源码。
   - 显示字符数、行数。

2. 中间：特征元素配置区
   - 由两个配置模块组成：
     - 变量名
     - 字符串字面量
   - 每个模块都支持：
     - 动态新增一项
     - 删除指定项
     - 输入框逐项填写
   - 表单只接受文本输入，不提供 JSON 模式切换。

3. 右侧：结果输出区
   - 展示处理后的压缩结果。
   - 只读。
   - 支持复制。
   - 显示字符数、行数和处理状态。

桌面端采用 `源码区 + 配置区 + 结果区` 三列布局，其中源码区和结果区较宽，配置区较窄。移动端按 `源码 -> 配置 -> 结果` 纵向堆叠。

## 页面交互

### 顶部说明

模块名称可以继续保留为 `JS 代码混淆`，但副标题必须明确真实语义：

`整体仅压缩，按配置定向处理变量名与字符串字面量。`

这样可避免用户误解为整段变量名和控制流都会被自动混淆。

### 操作按钮

保留三种核心动作：

1. `开始处理`
   - 触发后端定向改写与压缩流程。
   - 如果源码为空则禁用。

2. `清空`
   - 清空源码、配置项和结果。

3. `复制结果`
   - 仅当结果存在时可用。

### 表单行为

配置区的两组列表都按以下规则工作：

1. 每组默认至少显示一行空输入框，降低第一次使用门槛。
2. 用户可以添加多行，也可以删除任意一行。
3. 前端提交前会清洗数据：
   - 去掉首尾空白
   - 去掉空项
   - 去重
4. 如果某组清洗后为空，表示该组不参与定向处理。

### 处理中的状态

当用户点击 `开始处理` 后：

1. 源码输入区锁定。
2. 配置区锁定。
3. 清空按钮禁用。
4. 主按钮显示处理中状态。
5. 结果区先清空旧结果，再等待新结果返回。

这样可以避免请求期间继续编辑导致“结果与提交内容不一致”的状态错位。

## 数据模型

页面提交到后端的数据结构为：

```ts
interface JsTargetedTransformRequest {
  sourceCode: string;
  identifierTargets: string[];
  stringLiteralTargets: string[];
}
```

后端返回的数据结构为：

```ts
interface JsTargetedTransformResponse {
  outputCode: string;
}
```

错误响应统一为：

```ts
interface JsTargetedTransformErrorResponse {
  error: string;
}
```

## 处理流程

后端处理必须分为两个阶段，而不是调用整段混淆库一次完成。

### 阶段一：AST 定向改写

将用户输入的源码解析为 AST，只对两类节点做定向处理：

1. 变量名
   - 仅处理用户配置中列出的变量名。
   - 未配置的变量名保持原样。
   - 不做全局批量重命名。

2. 字符串字面量
   - 仅处理用户配置中列出的字符串字面量。
   - 未配置的字符串保持原样。

这里的“处理”不是整段混淆，而是生成稳定的替代值，再把 AST 中命中的节点替换掉。替代值应满足：

1. 可执行。
2. 不再保留原始可读文本。
3. 同一个特征元素在同一次处理内映射结果一致。

### 阶段二：整体压缩输出

在 AST 定向改写完成后，对生成结果做统一压缩。压缩目标是：

1. 删除多余空白。
2. 合并可安全合并的格式结构。
3. 尽量输出紧凑结果。

压缩阶段不能额外引入：

1. 整文件变量重命名。
2. 属性名混淆。
3. 控制流改写。

因此，“压缩”和“定向处理”是两个边界清晰的阶段：

- 压缩负责整体体积与格式收敛。
- AST 负责局部特征元素替换。

## 定向处理规则

### 变量名规则

1. 只有变量名命中 `identifierTargets` 时才会被改写。
2. 未命中的变量名保持原样。
3. 对象属性名不参与此规则，即使文本一致也不应被误改。
4. 不能因为命中了一个变量名，就把整文件所有同名文本做字符串替换。
5. 同一个变量名在一次处理中的替代结果必须一致。

### 字符串字面量规则

1. 只有字符串字面量命中 `stringLiteralTargets` 时才会被改写。
2. 未命中的字符串保持原样。
3. 同一个字符串字面量在一次处理中的替代结果必须一致。
4. 字符串改写后必须仍然生成合法的 JS 表达式。

### 缺失命中规则

如果用户在配置中填写了源码里不存在的变量名或字符串：

1. 本次处理仍然成功。
2. 不报错。
3. 该项仅视为“未命中”。

第一版不额外展示未命中统计。

## 实现建议

该需求更适合 AST 工具链，而不是现有的整段混淆库。

推荐实现方式：

1. 使用 AST 解析、遍历、生成工具完成定向改写。
2. 使用独立压缩工具完成最终压缩。

推荐原因：

1. 能精确区分变量名、字符串字面量和对象属性名。
2. 能严格遵守“只改指定元素”的边界。
3. 后续如果新增“函数名”一类特征元素，也可以在 AST 层自然扩展。

现有整段混淆依赖不适合作为核心实现，因为它天然倾向全局处理，不符合本次“局部定向处理”的约束。

## 错误处理

### 前端错误

1. 源码为空：禁止提交，并提示用户先输入源码。
2. 配置为空：允许提交，此时仅执行整体压缩。
3. 复制失败：提示用户手动复制，不影响结果保留。

### 后端错误

1. 源码无法解析：返回明确错误，例如“JS 语法解析失败”。
2. AST 改写失败：返回明确错误。
3. 压缩失败：返回明确错误。

错误发生时：

1. 前端保留源码输入。
2. 前端保留配置输入。
3. 不自动清空用户数据。
4. 结果区维持空状态或最新失败前的清空状态。

## 文件影响范围

现有实现已经存在，因此本次变更属于“能力重构”，不是新增平行模块。预计影响以下区域：

1. 页面层
   - `src/app/(root)/tools/js-obfuscator/page.tsx`
   - 配置区可能拆出独立组件

2. API 层
   - `src/app/api/tools/js-obfuscator/route.ts`

3. 核心处理层
   - `src/lib/tools/js-obfuscator.ts`
   - 需要从“整段混淆”改为“AST 定向改写 + 压缩输出”

4. 测试层
   - `src/lib/tools/__tests__/js-obfuscator.test.ts`
   - 需要重写为覆盖新语义

## 测试策略

测试重点如下：

1. 空源码时报错。
2. 无配置时只做整体压缩。
3. 指定变量名会被改写。
4. 未指定变量名保持原样。
5. 指定字符串字面量会被改写。
6. 未指定字符串字面量保持原样。
7. 对象属性名不会被误改。
8. 结果代码仍然可以执行。
9. 配置中存在重复项时不会导致重复处理。
10. 配置中包含未命中项时，整体处理仍成功。

## 成功标准

本次改造完成后，应满足以下标准：

1. 页面为三段式布局：源码、配置、结果。
2. 配置区通过表单输入变量名和字符串字面量。
3. 未配置的变量名和字符串不被误改。
4. 对象属性名不被误改。
5. 输出结果整体被压缩。
6. 结果代码仍可执行。
7. 整体处理失败时，源码和配置不会被清空。
